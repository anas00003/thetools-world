# On-Page SEO and GEO Standards

## Title tag pattern

Use the task, size or format, and brand:

`Primary Task Online | TheTools.World`

Examples:

- `Compress JPG to 20KB Online | TheTools.World`
- `Resize Signature to 140x60 Online | TheTools.World`

## Meta description pattern

Use one concise sentence that names inputs, target outcome, use cases, and honesty where needed. Avoid stuffing variants.

## H1 rule

Use one H1 per page. Match the main user task in natural language. Do not use multiple keyword variants as headings.

## Intro copy rule

The first paragraph should say who the page is for, what setting is preselected, and what the user may need to check after download.

## Tool above fold rule

The tool UI should appear before long education content. On mobile, upload, target settings, and action button must be reachable without horizontal scrolling.

## Trust note rule

Use precise trust language only:

- Browser APIs are used for the selected image task.
- No account is required if that is true.
- Avoid closing the page until the download is ready.

Do not add broader privacy or server claims unless the implementation proves them.

## Exact-size honesty rule

Never promise exact output. Use wording such as:

- "toward a 50KB target"
- "close to or under the limit when possible"
- "exact final size depends on the original image, dimensions, detail, format, and quality"

## How-to section rule

Every tool page should include short steps:

1. Choose the file.
2. Confirm the default target or dimensions.
3. Adjust quality, width, format, or dimensions.
4. Download and check final file size or dimensions.
5. Compare with official upload rules.

## Troubleshooting section rule

Exact-size pages should explain what to do if the output is still too large:

- Reduce max width.
- Lower quality gradually.
- Resize first for large photos.
- Use a larger target page if the form allows it.
- Check format and dimension rules.

## FAQ section rule

Add FAQ only when it answers real user concerns. Keep it visible on the page. Do not use FAQ as a schema-only trick.

## Schema rule

Use `FAQPage` only when visible FAQ exists and schema text exactly matches visible questions and answers. Do not add ratings, reviews, aggregateRating, fake authors, or fake popularity claims.

## Internal linking rule

Link to existing live pages only. Use natural anchors such as:

- image compressor
- image resizer
- compress image to 20KB
- compress image to 50KB
- resize image for online form

Do not link to future pages until they are built.

## Mobile UX rule

Controls must stack cleanly. Inputs, file pickers, buttons, and result boxes must fit within the viewport. Avoid decorative layouts that push the tool too far down.

## Core Web Vitals rule

Keep pages static and lightweight. Avoid third-party scripts until there is a clear reason. Do not add AdSense or analytics changes inside tool implementation prompts unless explicitly approved.

## GEO and AI search rule

Write clear factual sections that answer the user task directly. Use concise headings, exact tool limitations, and practical steps. Do not invent statistics, authority claims, or official-form advice.

## Things never to do

- Do not claim every image can hit an exact size.
- Do not add fake testimonials, fake reviews, fake ratings, fake user counts, or "#1" claims.
- Do not keyword stuff headings or body copy.
- Do not create thin pages for every keyword variant.
- Do not add links to pages that do not exist.
- Do not add AdSense code during page creation unless the task explicitly approves it.
- Do not change canonical or domain strategy without a dedicated deployment prompt.
